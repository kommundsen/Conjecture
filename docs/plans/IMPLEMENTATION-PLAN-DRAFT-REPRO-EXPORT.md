# Draft: Minimal Reproducible Example Export

## Motivation

When a property test fails, the developer needs to communicate that failure — to teammates, in bug reports, in CI logs, or to downstream teams who may not use Conjecture. Today, the failure output includes the shrunk counterexample values and a seed for reproduction, but reproducing requires Conjecture installed and the same test code. Exporting a standalone `.cs` file with hard-coded values and the failing assertion makes the failure self-contained, portable, and archivable.

## .NET Advantage

.NET's strong typing and `System.Text.Json` serialization make it straightforward to emit a compilable C# file that reconstructs the counterexample. Roslyn's `SyntaxFactory` (or simpler template-based generation) can produce correctly formatted, compilable code. The exported file can target any test framework (xUnit, NUnit, MSTest) or be a plain console app — no Conjecture dependency needed.

## Key Ideas

### Automatic Export on Failure
```csharp
new ConjectureSettings
{
    ExportReproOnFailure = true,
    ReproOutputPath = "test-failures/",
}
```

When a property fails, Conjecture writes a file like:

```csharp
// Auto-generated by Conjecture on 2026-04-06
// Property: AdditionTests.CommutativeProperty
// Seed: 0xABCD1234
// Shrunk from 47 examples to minimal counterexample

using Xunit;

public class CommutativeProperty_Repro
{
    [Fact]
    public void Counterexample()
    {
        // Minimal counterexample values
        int a = int.MaxValue;
        int b = 1;

        // Original property assertion
        var result = a + b;
        Assert.Equal(a + b, b + a); // This fails due to overflow
    }
}
```

### Complex Type Serialization
```csharp
// For types with constructors:
var person = new Person("Alice", -1);

// For types requiring JSON deserialization:
var order = JsonSerializer.Deserialize<Order>("""
{
    "Id": 42,
    "Items": [{"Name": "Widget", "Quantity": 0}],
    "Total": -5.99
}
""");
```

### Stateful Test Reproduction
```csharp
// State machine failures export the full command sequence
public class CounterMachine_Repro
{
    [Fact]
    public void Counterexample()
    {
        var sut = new Counter();

        sut.Increment();  // State: 1
        sut.Increment();  // State: 2
        sut.Decrement();  // State: 1
        sut.Decrement();  // State: 0
        sut.Decrement();  // State: -1 — invariant violated: count should never be negative
    }
}
```

### Export Formats
- **Standalone xUnit `[Fact]`** (default) — drop into any xUnit project
- **Standalone NUnit `[Test]`** — for NUnit users
- **Standalone MSTest `[TestMethod]`** — for MSTest users
- **Console app** — `dotnet run` to reproduce, no test framework needed
- **JSON data file** — just the counterexample values, for consumption by other tools

### CLI Integration
```bash
# Export the last failure
dotnet conjecture export --last --output repro.cs

# Export all failures from the example database
dotnet conjecture export --all --output-dir test-failures/

# Export as console app
dotnet conjecture export --last --format console --output repro.cs
```

### Value Rendering
- Primitives: literal syntax (`42`, `"hello"`, `true`, `3.14`)
- Collections: collection expressions (`[1, 2, 3]`) or initializer syntax
- Records: constructor call (`new Person("Alice", 30)`)
- Complex objects: JSON deserialization fallback
- Enums: qualified name (`Color.Red`)
- Nullable: `null` or value
- Custom: `IReproFormatter<T>` extensibility point

## Design Decisions to Make

1. Should export be automatic on failure or manual via CLI/API?
2. How to handle types that can't be easily constructed in standalone code? (Private constructors, complex dependency graphs)
3. Should the exported file include the original property body or just the assertion?
4. How to handle `async` properties in the export?
5. Framework detection: auto-detect which test framework to target, or always require explicit choice?
6. Should exports include the Conjecture seed as a comment for cross-reference?
7. Code generation: use Roslyn `SyntaxFactory` (correct but heavy dependency) or string templates (simpler but fragile)?

## Scope Estimate

Medium. Core value rendering + template generation is ~2 cycles. Stateful test export, CLI integration, and format support add ~1-2 more.

## Dependencies

- Existing `CounterexampleFormatter` and `StateMachineFormatter`
- `System.Text.Json` for complex type serialization
- `ConjectureSettings` for configuration
- Existing example database for historical failure access
- Optional: Roslyn `SyntaxFactory` for robust code generation

## Open Questions

- Should exports be version-controlled? (Add to `.gitignore` or commit as regression tests?)
- How to handle non-serializable types (streams, database connections, disposables)?
- Should the export include setup/teardown code if the property has dependencies?
- Can we generate a `.csproj` alongside the `.cs` for truly standalone reproduction?
- Should the MCP tool (`explain-shrink-output`) integrate with this to provide AI-readable failure descriptions?
- How to handle platform-specific failures? (Export should note the OS, .NET version, architecture)
