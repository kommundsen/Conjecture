# Messaging property testing

Conjecture's messaging support — `Conjecture.Messaging` and the satellite adapter packages — lets you write property tests against message-queue and service-bus systems the same way you write them against HTTP endpoints. This page explains the model: why the abstractions look the way they do, what the shrinker does for you, and what trade-offs you're signing up for.

## The two-layer model

Messaging support sits on top of the v0.20 [transport-agnostic interaction foundation](../decisions/0059-conjecture-interactions-and-http-architecture.md) (`Conjecture.Interactions`):

- **Layer 1** — `IInteraction`, `IInteractionTarget`, `InteractionStateMachine<TState>`. Knows nothing about messaging or HTTP. The shrinker and state-machine machinery live here.
- **Layer 2** — `Conjecture.Messaging`. Adds a `MessageInteraction` record and `IMessageBusTarget` interface that extend the Layer-1 contracts with messaging semantics: publish, receive, ack, nack.
- **Layer 3** — concrete broker adapters: `Conjecture.Messaging.AzureServiceBus`, `Conjecture.Messaging.RabbitMq`. Each ships as its own NuGet package so a project that only needs the in-memory adapter doesn't take a transitive dependency on `Azure.Messaging.ServiceBus` or `RabbitMQ.Client`.

The same `Property.ForAll(target, strategy, assertion, ct)` shape works against any of them. When you switch from the in-memory adapter to a real broker, you change the target reference and nothing else.

## What `MessageInteraction` carries

```csharp
public sealed record MessageInteraction(
    string Destination,
    ReadOnlyMemory<byte> Body,
    IReadOnlyDictionary<string, string> Headers,
    string MessageId,
    string? CorrelationId = null) : IInteraction;
```

- `Destination` is the queue or topic name. `CompositeInteractionTarget` from Layer 1 routes by destination, so a single test can fan out across multiple resources.
- `Body` is `ReadOnlyMemory<byte>`. Every body strategy in Conjecture composes here — `Generate.FromProtobuf<T>()`, `Generate.FromJsonSchema(schema)`, plain `Generate.Bytes(...)`. No per-format glue is needed.
- `MessageId` is generated by the strategy, **not** by the broker. This is the load-bearing decision: the shrinker needs to replay sequences byte-for-byte, and broker-assigned IDs are non-deterministic. `Generate.Messaging.Publish(...)` synthesizes deterministic `MessageId`s under the same seed.
- `Headers` round-trip strings. Brokers that use byte-array values (RabbitMQ) encode/decode UTF-8 automatically. Non-string values from broker headers (Azure Service Bus' `ApplicationProperties` allows `object`) are converted via `Convert.ToString(value, CultureInfo.InvariantCulture)` rather than silently dropped.
- `CorrelationId` is nullable — set it for request/reply or saga flows, leave it null otherwise.

## Why pull-based receive (not push)

`IMessageBusTarget` exposes a pull-based receive contract:

```csharp
Task<MessageInteraction?> ReceiveAsync(string destination, TimeSpan timeout, CancellationToken ct);
Task AcknowledgeAsync(MessageInteraction message, CancellationToken ct);
Task RejectAsync(MessageInteraction message, bool requeue, CancellationToken ct);
```

A push-based handler-registration model (`target.OnReceive(destination, handler)`) would be more familiar to users coming from `MassTransit` or `NServiceBus`, but it doesn't compose with the shrinker. `CommandSequenceShrinkPass` from `Conjecture.Core` requires each step in a recorded interaction sequence to return synchronously to the shrinker. Push handlers fire on broker threads at non-deterministic times, which means the shrinker can't replay a sequence that depends on push timing.

Pull-based receive plus explicit ack/reject also lets you property-test the failure modes you most want to catch: redelivery on `RejectAsync(requeue: true)`, dead-lettering on `RejectAsync(requeue: false)`, and the order in which uncommitted messages become visible after a consumer crash.

## How shrinking handles publish/consume pairs

When a property fails on a sequence like `[Publish("a", body1), Consume("a"), Publish("b", body2), Consume("b"), Publish("a", body3)]`, the shrinker tries to delete each step independently. The interesting invariant lives at the boundary:

> **A `Publish(d, body)` whose body is never read by any subsequent `Consume(d)` must be droppable in isolation. A `Consume(d)` whose result is never asserted on must also be droppable in isolation. The pair is droppable together when neither side has observable downstream effects.**

You don't need to write any code for this. `CommandSequenceShrinkPass` already attempts every individual deletion; the in-memory adapter's deterministic semantics make the "no observable effect" check exact. Real-broker shrinking inherits the same property when run against a deterministic broker mode (the Azure Service Bus emulator, a fresh RabbitMQ container with persistence disabled).

## The two-tier test plan

Each concrete adapter ships with two tiers of tests:

| Tier | Where | Speed | Docker | Purpose |
|---|---|---|---|---|
| Unit | `Conjecture.Messaging.<Broker>.Tests/` | ms | No | Adapter logic — translation between `MessageInteraction` and the broker SDK's native types, ack/nack routing, lifecycle. Hand-rolled fakes for the seam interfaces (`IServiceBusClientAdapter`, `IRabbitMqConnectionAdapter`, etc.). |
| Integration | `Conjecture.SelfTests/Messaging/<Broker>/` | seconds (container startup) | Yes (Testcontainers) | Real-broker drift — confirms the SDK still behaves the way the adapter expects after package upgrades. Class-scoped fixture, gated on Docker availability so the inner loop on a no-Docker dev machine stays fast. |

The unit tier runs on every `dotnet test`. The integration tier is gated behind `[Trait("RequiresDocker", "true")]` and skipped silently when no Docker is available. The fakes carry ~95% of adapter logic; the integration tier guards against silent SDK semantic changes.

## What you give up

Pull-based receive forces users with push-based consumer code to add a polling loop in their tests. The test loop is already a polling shape (`await Property.ForAll(...)`), so the cost is one extra `await` per consume — but if your production consumer uses `IHostedService` background loops, your test code won't look identical.

Cross-broker test portability is per-package. Moving a test from RabbitMQ to Azure Service Bus requires swapping the target and may surface header-key conventions that the abstraction doesn't paper over (RabbitMQ's `x-routing-key` has no exact ASB equivalent).
