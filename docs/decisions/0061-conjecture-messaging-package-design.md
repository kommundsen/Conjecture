# 0061. Conjecture.Messaging package design

**Date:** 2026-04-25
**Status:** Accepted

## Context

ADR 0059 ships `Conjecture.Interactions` (`IInteraction`, `IInteractionTarget`, `InteractionStateMachine`, `CompositeInteractionTarget`) plus an HTTP concretion (`Conjecture.Http`) as the v0.20 interaction foundation. Layer 1 was kept deliberately minimal until a second non-HTTP transport actively consumed it, at which point its contract could be reviewed.

Messaging is the natural first non-HTTP transport: queue / topic semantics (publish, consume, in-flight ordering, redelivery, dead-letter) are sufficiently different from HTTP request/response that they stress-test the abstraction before user-facing host packages (AspNetCore v0.24, Aspire v0.25) start depending on it. Service-bus property testing is also independently high-value — race-condition bugs in consumers are notoriously hard to catch with example-based tests.

The decision must answer:

- What does a `MessageInteraction` carry?
- How does `IMessageBusTarget` model consumption — pull or push?
- How are publish/consume pairs shrunk when one side never observes the other?
- How do concrete adapters ship — in the core package or as separate sub-packages?
- How do body strategies compose with the v0.21 schema-driven generators?
- What is the default broker fixture scope?

## Decision

Ship `Conjecture.Messaging` (transport-agnostic primitives + in-memory reference adapter) plus two satellite packages for concrete brokers.

### Layer 1 — `Conjecture.Messaging` (broker-agnostic)

**`MessageInteraction : IInteraction`** — immutable record:

```csharp
public sealed record MessageInteraction(
    string Destination,
    ReadOnlyMemory<byte> Body,
    IReadOnlyDictionary<string, string> Headers,
    string MessageId,
    string? CorrelationId = null) : IInteraction;
```

- `Destination` is the resource name routed by `CompositeInteractionTarget` (queue or topic name; broker-agnostic).
- `Body` is `ReadOnlyMemory<byte>` so JSON / protobuf / arbitrary payload strategies share one shape.
- `MessageId` is required and supplied by the strategy (deterministic for shrinking — never generated by the broker at publish time).
- `CorrelationId` is nullable: only set for messages participating in a request/reply or saga flow.

**`IMessageBusTarget : IInteractionTarget`** — extends Layer 1 with a pull-based consume hook:

```csharp
public interface IMessageBusTarget : IInteractionTarget
{
    Task<MessageInteraction?> ReceiveAsync(string destination, TimeSpan timeout, CancellationToken ct);
    Task AcknowledgeAsync(MessageInteraction message, CancellationToken ct);
    Task RejectAsync(MessageInteraction message, bool requeue, CancellationToken ct);
}
```

`ExecuteAsync` from `IInteractionTarget` handles publishes (the `IInteraction` is a `MessageInteraction`); `ReceiveAsync` / `AcknowledgeAsync` / `RejectAsync` cover consumption. Pull-based consumption is chosen over push-based handler registration because:

- Property tests need deterministic stepping — `await ReceiveAsync(...)` lets the test loop interleave publishes and consumes in a single sequence the shrinker can replay.
- Push-based handler registration forces a callback model that doesn't compose with `InteractionStateMachine<TState>`, where each step must return synchronously to the shrinker.
- Real-world brokers (Azure Service Bus, RabbitMQ) all expose pull semantics underneath — push handlers are conveniences built on top.

**`InMemoryMessageBusTarget : IMessageBusTarget`** — reference adapter ships in the same package. Backed by per-destination `Channel<MessageInteraction>` so it inherits .NET's well-tested in-memory queue semantics. Used by the package's own self-tests and by user tests that don't need a real broker.

### Shrinking semantics for publish/consume pairs

Publishes and consumes are independent steps in an `InteractionStateMachine` sequence. The shrinker — driven by `CommandSequenceShrinkPass` — already deletes individual steps; we add one extra invariant:

> **A `Publish(d, body)` whose body is never read by any subsequent `Consume(d)` must be droppable in isolation. A `Consume(d)` whose result is never asserted on must also be droppable in isolation. The pair is droppable together when neither side has observable downstream effects.**

This is enforced by `CommandSequenceShrinkPass` already (it tries deleting each step and accepts shrinks that still reproduce the failure). No new shrinking machinery is needed; `InMemoryMessageBusTarget`'s deterministic semantics make the "no observable effect" check exact for the in-memory case. Real brokers shrink against their own deterministic in-memory mode (Azure Service Bus emulator, RabbitMQ test container with persistence disabled).

### Layer 2 — Concrete adapter sub-packages

- **`Conjecture.Messaging.AzureServiceBus`** — `AzureServiceBusTarget : IMessageBusTarget` over `Azure.Messaging.ServiceBus`. Resource names map to queue / topic names within a single `ServiceBusClient`.
- **`Conjecture.Messaging.RabbitMq`** — `RabbitMqTarget : IMessageBusTarget` over `RabbitMQ.Client`. Resource names map to queue names; topic-style routing is exposed via the `Headers` dictionary (`x-routing-key`).

Both ship as separate NuGet packages so users who only need the in-memory adapter (or only one broker) don't take a transitive dependency on the other.

### Body strategy composition

`MessageInteraction.Body` is `ReadOnlyMemory<byte>` so any byte-producing strategy composes:

- `Generate.FromProtobuf<T>()` (v0.21) → `Strategy<ReadOnlyMemory<byte>>`.
- `Generate.FromJsonSchema(schema)` (v0.21) returns JSON; wrap in `.Select(json => Encoding.UTF8.GetBytes(json))`.
- Arbitrary `Strategy<byte[]>` from `Generate.Bytes(...)` lifts via `.Select(b => (ReadOnlyMemory<byte>)b)`.

`Generate.Messaging.Publish(destination, bodyStrategy)` (sub-issue #464) takes the body strategy as its sole content parameter and fills in `MessageId` from a deterministic `Generate.Guid()` sequence so shrinks are reproducible.

### Per-adapter fixture scope

Class-scoped by default, mirroring the v0.20 fixture seam (ADR 0059, "Deferred per-adapter fixture wiring"). One broker connection / container per test class amortizes the (significant) cost of bringing up a real broker while keeping per-test isolation cheap (each test gets a fresh queue). Test-scoped or assembly-scoped fixtures are opt-in via the same `ConjectureFixtureBase` seam used by HTTP host fixtures.

## Consequences

**Easier:**
- Same `Property.ForAll(target, strategy, assertion, ct)` shape as HTTP — users who learned the HTTP transport need no new vocabulary.
- Pull-based consumption composes naturally with `InteractionStateMachine` and `CommandSequenceShrinkPass` — no new shrinking code.
- Body byte-arrays compose with all v0.21 schema generators without per-format glue.
- Concrete adapters can be added later (Kafka, NATS, AWS SNS/SQS) without touching `Conjecture.Messaging` itself — they only need to implement `IMessageBusTarget`.

**Harder:**
- Pull semantics force users with push-based consumer code to add a polling loop in their tests. Acceptable: the test loop is already a polling shape (`await Property.ForAll(...)`).
- Cross-broker test portability is per-package — moving from RabbitMQ to Azure Service Bus requires swapping the target and may surface header-key differences that the abstraction doesn't paper over.

**Risks:**
- `IMessageBusTarget`'s consume contract is wider than `IInteractionTarget`. If a third transport (gRPC streaming, SignalR) wants similar pull semantics, we may want to lift `ReceiveAsync` / `AcknowledgeAsync` to a shared `IPullTarget` interface. Defer until that third transport actually exists; matches ADR 0059's "minimal until needed" policy.
- Real-broker shrinking depends on each broker's emulator / deterministic test mode. Azure Service Bus emulator and RabbitMQ test containers are both reliable in 2026, but a future broker without a deterministic mode would degrade shrinking quality.

### Per-adapter test strategy

Concrete adapter test suites split into two tiers to keep the dev inner loop fast while still catching real-broker drift:

- **Unit tier** (`Conjecture.Messaging.<Broker>.Tests`) — hand-rolled fakes for the broker SDK's client/sender/receiver types. No Docker, no network, runs in milliseconds. Covers the adapter's translation between `MessageInteraction` and the broker's native message type, error mapping, and lifecycle ordering. This is the tier that runs on every `dotnet test` and in every PR check.
- **Integration tier** (`Conjecture.SelfTests/Messaging/<Broker>/`) — runs against the real broker (Azure Service Bus emulator via Docker for ASB; the official RabbitMQ Docker image for RabbitMQ). Class-scoped fixture per broker — startup cost is amortized across all tests in the class. Gated on `DOCKER_HOST` availability (or equivalent broker-connection-string env var) and skipped silently when unavailable, so the local inner loop on a no-Docker dev machine stays fast. Runs on CI nightly and on PRs labelled `integration`.

This is deliberately leaner than mandating Testcontainers everywhere — Testcontainers' 5–10 s container startup is unbearable at the unit-test tier. The fake-client tier covers ~95% of adapter logic; the integration tier guards against SDK upgrades silently changing semantics.

## Alternatives Considered

**Push-based handler registration** — `target.OnReceive(destination, handler)`. Rejected: forces a callback model incompatible with `InteractionStateMachine`'s step-by-step shrinking. The shrinker needs to interleave publishes and receives in one deterministic sequence; callbacks fire on broker threads at non-deterministic times.

**Single unified `Conjecture.Messaging` with all broker adapters bundled** — one NuGet package containing the in-memory, Azure Service Bus, and RabbitMQ adapters. Rejected: forces a 5+ MB transitive dependency on `Azure.Messaging.ServiceBus` + `RabbitMQ.Client` even when only the in-memory adapter is needed. ADR 0055 (extension blocks for satellite packages) already establishes the per-broker sub-package pattern.

**Typed body** — `MessageInteraction<TBody>` generic with a deserializer parameter on `IMessageBusTarget`. Rejected: state-machine shrinking requires a uniform interaction shape (ADR 0059's "Generic `IInteractionTarget<TResult>`" alternative was rejected for the same reason). Body bytes plus a `ContentType` header (when needed) keeps the shape uniform.

**`MessageId` generated by the broker** — let the broker assign IDs at publish time. Rejected: shrinks must be deterministic. A strategy-supplied `MessageId` (from `Generate.Guid()` over a fixed seed) makes failed runs reproducible byte-for-byte.

**Test-scoped broker fixture by default** — fresh broker container per test. Rejected: container startup cost (1–3 s for RabbitMQ, 5–10 s for Service Bus emulator) makes a 100-test property suite unbearable. Class-scoped amortizes the cost; per-test isolation is preserved by giving each test a fresh queue / topic name.
